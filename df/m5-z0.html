faithful firefly posted at 2021-07-19T04:00:11.000Z

Actually, -4 is fc h (space added for clarity that h is not a digit). It is only two hex digits, or more accurately one byte. The SBYTE means it is signed using two's complement. movsx is the sign-extended mov operation. Meaning it extends the most significant bit to fill all of the bits in the register that aren't taken up with the initial value

var1 points to -4 (and if you add some offsets, you can point to other bytes but that's besides the point.)

var1 = fc h

edx is an extended register meaning it's 32 bits or four bytes or eight hex chars: ???????? h (we don't know edx's value)

mov edx,var1 would not work because their sizes don't match( 8 bits != 32 bits)

so movsx is used

movsx edx,var1 ; edx = ??????fc h

But movsx also has the additional behavior that it extends the sign bit (the highest bit of var1, which is 1) across the rest of the register.

So it actually looks like this:

movsx edx,var1 ; edx = fffffffc h

and so even though edx is a 32 bit two's complement number, edx still = -4 because of the padding.

advanced anaconda posted at 2021-07-19T04:54:27.000Z

"The MOVSX instruction (move with sign-extend) copies the contents of a source operand into a destination operand and sign-extends the value to 16 or 32 bits. "

-4 is SBYTE, so it's a one-byte value fch, or 11111100.

The destination register edx is of length 8 bytes. 

So the MOVSX instruction extends the sign bit of fch, which is 1 to fit the gap on the left. Therefore, you get 1s before 11111100 to make it 32 bits. So you end up with fffffffch in hexadecimal.
